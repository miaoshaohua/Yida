## 一、项目概况

### 1.1 项目简介
AI虚拟试衣间（Yida），用户上传人物照片和服装照片，调用FASHN VTON API生成AI试衣效果。包含C端用户应用和B端管理后台。

### 1.2 技术栈
| 层级 | 技术 | 版本 |
|------|------|------|
| 前端框架 | React + TypeScript | 18.3.1 / 5.2.2 |
| 构建工具 | Vite | 5.2.0 |
| 路由 | React Router DOM | 6.23.0 |
| HTTP客户端 | Axios | 1.6.8 |
| 后端框架 | NestJS | 10.x |
| ORM | Prisma | 5.x |
| 数据库 | SQLite（开发）/ PostgreSQL（生产） | - |
| 任务队列 | Bull + Redis | - |
| 对象存储 | Cloudflare R2（S3兼容） | - |
| API文档 | Swagger | - |

## 二、踩坑记录与解决方案

### 2.1 环境变量占位符问题 🔴 严重

**问题**：`.env`文件中R2配置使用了占位符值：
```
R2_ACCOUNT_ID=your-r2-account-id
R2_ACCESS_KEY_ID=your-r2-access-key-id
R2_SECRET_ACCESS_KEY=your-r2-secret-access-key
```

**表现**：
- 预签名URL生成成功，但域名是`https://yida-images.your-r2-account-id.r2.cloudflarestorage.com`
- 前端上传时报错：`net::ERR_SSL_VERSION_OR_CIPHER_MISMATCH`
- 因为域名不存在，SSL握手失败

**根因**：AI生成代码时使用了示例占位符，但没有提醒用户必须替换为真实值，也没有实现降级方案。

**解决方案**：
1. 在`.env.example`中明确标注哪些是必填项
2. StorageService启动时检测R2配置是否为占位符，如果是则自动降级到本地存储
3. 前端上传时，如果R2不可用，走后端`/api/storage/upload`接口上传到本地

**经验规则**：
> ⚠️ 环境变量中所有包含`your-`前缀的值都是占位符，必须替换为真实值或实现降级方案。AI在生成配置时，应该同时生成配置检测和降级逻辑。

---

### 2.2 后端路由前缀不一致 🔴 严重

**问题**：后端添加了全局前缀`api`（`app.setGlobalPrefix('api')`），但新增Controller时没有考虑到这一点。

**表现**：
- 前端调用`POST /api/storage/presigned-url`，后端报错`Cannot POST /api/storage/presigned-url`
- 实际上是因为后端代码变更后没有重启服务，路由没有注册

**根因**：两个问题叠加：
1. 使用`npm run start`而非`npm run start:dev`启动后端，代码变更不自动重载
2. 没有在变更后验证路由是否生效

**解决方案**：
1. 开发环境使用`npm run start:dev`启动，支持热重载
2. 每次新增/修改路由后，用PowerShell或Swagger验证：
   ```powershell
   Invoke-RestMethod -Uri "http://localhost:3000/api/your-route" -Method POST -Body '{}' -ContentType "application/json"
   ```
3. 访问Swagger文档确认路由已注册：`http://localhost:3000/api/docs`

**经验规则**：
> ⚠️ 后端全局前缀为`api`，Controller中不需要再加`/api`。代码变更后必须确认服务已重载，用API测试验证路由生效。

---

### 2.3 前端API路径重复 🟡 中等

**问题**：前端api.ts中axios实例已设置`baseURL: '/api'`，但有时会误写完整路径。

**正确写法**：
```typescript
api.post('/storage/presigned-url', { filename })  // ✅ 正确
```

**错误写法**：
```typescript
api.post('/api/storage/presigned-url', { filename })  // ❌ 会导致 /api/api/storage/presigned-url
```

**经验规则**：
> ⚠️ 前端axios baseURL已设为`/api`，调用时不要再加`/api`前缀。Vite代理会将`/api/*`转发到`http://localhost:3000/api/*`。

---

### 2.4 页面原型与实现不一致 🟡 中等

**问题**：开发出的前端页面样式与`documents/页面原型设计.html`不一致。

**表现**：
- 首页缺少上传区域和分类选择
- 没有底部导航栏
- 颜色和圆角不符合设计规范
- 登录页布局与原型差异大

**根因**：AI在实现页面时，没有主动对照原型文档，而是根据文字描述自行发挥。

**解决方案**：
1. 实现页面时，先用浏览器打开`documents/页面原型设计.html`查看设计
2. 严格按照原型的布局、颜色、间距实现
3. 实现后截图对比，确认一致性

**关键设计规范**：
- 品牌渐变：`linear-gradient(135deg, #E6004C 0%, #FF2A6D 100%)`
- 背景渐变：`linear-gradient(180deg, #FFF5F8 0%, #FFE8EF 100%)`
- 上传区域：虚线边框 + 浅粉背景
- 大圆角：24px（卡片）、16px（按钮）、12px（小元素）
- 底部导航栏：固定定位，当前页高亮

**经验规则**：
> ⚠️ 实现UI页面时，必须先读取原型文档，严格按照原型实现。不要自行发挥设计。

---

### 2.5 服务重启问题 🟡 中等

**问题**：修改后端代码后，服务没有自动重载，导致变更不生效。

**根因**：使用`npm run start`（生产模式）而非`npm run start:dev`（开发模式）。

**解决方案**：
1. 开发环境始终使用`npm run start:dev`
2. 如果必须手动重启，先停止所有Node进程：
   ```powershell
   taskkill /F /IM node.exe
   ```
3. 然后重新启动服务

**经验规则**：
> ⚠️ 开发时使用`npm run start:dev`启动后端，确保热重载。手动重启时先杀掉所有Node进程。

---

### 2.6 游客登录404错误 🟡 中等

**问题**：游客登录接口返回404。

**根因**：后端代码变更后没有重启服务，新添加的guest-login路由没有注册。

**解决方案**：重启后端服务，用API测试验证。

**经验规则**：
> ⚠️ 新增接口后，必须重启服务并验证路由可用，再在前端调用。

## 三、架构设计经验

### 3.1 双认证体系设计 ✅ 好的实践

项目采用了用户端和管理端分离的认证体系：
- 用户端：JwtAuthGuard + JwtStrategy + @CurrentUser()
- 管理端：AdminJwtAuthGuard + AdminJwtStrategy + @CurrentAdmin()

这种设计避免了权限混淆，管理端和用户端的token互不通用。

### 3.2 存储服务双模式 ✅ 好的实践

StorageService同时支持：
- Cloudflare R2（S3兼容）：预签名URL直传
- 本地文件系统：后端上传

但降级逻辑不够完善，R2配置为占位符时应该自动降级到本地存储。

### 3.3 任务队列设计 ✅ 好的实践

使用Bull + Redis实现异步任务队列：
- 试衣任务创建后进入队列
- TryOnProcessor异步处理
- 前端轮询任务状态

这种设计避免了长时间HTTP请求阻塞。

### 3.4 前端样式方案 ⚠️ 需要改进

当前使用inline style（JavaScript对象），存在以下问题：
- 无法使用CSS伪类（:hover, :active等）
- 无法使用CSS动画（@keyframes需要在CSS文件中定义）
- 代码冗长，可读性差

**建议**：后续项目考虑使用CSS Modules或Tailwind CSS。

## 四、AI开发流程经验

### 4.1 上下文管理 🔑 关键

**问题**：长对话导致上下文溢出，AI遗忘早期信息。

**解决方案**：
1. 当对话超过一定长度时，主动建议用户拆分任务
2. 将关键信息写入`.trae/rules/project_rules.md`，新对话自动读取
3. 拆分任务时，在文档中记录当前进度和待办事项

**最佳实践**：
- 一个对话处理1-2个功能模块
- 复杂功能拆分为：设计→实现→测试→修复
- 每次对话开始时，先读取项目规则文件

### 4.2 验证驱动开发 🔑 关键

**问题**：AI写完代码后没有验证，导致问题累积。

**解决方案**：
1. 每个功能实现后，立即启动服务验证
2. 后端接口用PowerShell或Swagger测试
3. 前端页面用浏览器查看
4. 发现问题立即修复，不要累积

**最佳实践**：
```
实现功能 → 启动服务 → 测试验证 → 修复问题 → 再次验证
```

### 4.3 原型优先原则 🔑 关键

**问题**：AI实现UI时自行发挥，不对照原型。

**解决方案**：
1. 实现UI前，先读取原型文档
2. 严格按照原型的布局、颜色、间距实现
3. 实现后截图对比

**最佳实践**：
- 原型是契约，不是参考
- 先还原，再优化
- 颜色、间距、圆角必须精确匹配

### 4.4 配置完整性检查 🔑 关键

**问题**：AI生成配置时使用占位符，但不提醒用户。

**解决方案**：
1. 生成`.env`时，在占位符旁添加注释说明
2. 实现配置检测逻辑，启动时检查关键配置
3. 提供降级方案，配置缺失时使用默认值

**最佳实践**：
- 每个外部依赖都要有降级方案
- 启动时打印配置状态
- 文档中列出所有必填配置项

## 六、Trae AI开发最佳实践总结

### 6.1 项目初始化阶段
1. 先写PRD和技术方案，再开始编码
2. 在`.trae/rules/project_rules.md`中记录关键架构决策
3. 环境变量模板中标注必填项和默认值

### 6.2 功能开发阶段
1. 每个功能模块独立对话，避免上下文过长
2. 实现后立即验证，不要累积未验证的代码
3. UI实现必须对照原型文档
4. 新增接口必须测试后再前端调用

### 6.3 测试修复阶段
1. 全流程自测，按用户操作路径逐个验证
2. 发现问题立即修复并重新验证
3. 将踩坑记录追加到项目规则文件

### 6.4 上下文切换时
1. 将当前进度写入文档
2. 更新项目规则文件
3. 新对话开始时读取规则文件

## 七、给未来AI开发者的提醒

1. **读规则文件**：每次新对话开始，先读`.trae/rules/project_rules.md`
2. **看原型文档**：实现UI前，先看`documents/页面原型设计.html`
3. **查现有代码**：新增功能前，先搜索项目中是否已有类似实现
4. **验证再继续**：每步实现后验证，不要一口气写完再测
5. **记录踩坑**：发现新问题，追加到规则文件中
6. **主动降级**：外部依赖不可用时，提供本地降级方案
7. **注意前缀**：后端全局前缀`api`，前端baseURL已设`/api`
8. **热重载**：开发时用`start:dev`，确保代码变更自动生效
