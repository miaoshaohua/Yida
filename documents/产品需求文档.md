## 1. 项目概况

### 一句话摘要
开发一个基于响应式设计的AI虚拟试衣Web应用，支持用户登录、云端存储历史记录、付费会员体系，用户上传人物照片和衣服图片，调用API生成试衣效果并展示。

### 项目背景
**用户痛点**：
- 网购衣服无法上身效果不确定，退换货率高
- 线下试衣时间成本高，效率低
- 穿搭博主/商家需要快速预览搭配效果

**数据依据（参考）**：
- 服装电商退换货率约30%，其中70%因尺码/效果不满意
- AI虚拟试衣是近期用户兴趣指数同比增长300%

### 产品目标
- **目标1**：支持用户登录/注册，提供完整账户体系
- **目标2**：单次试衣流程控制在3步以内
- **目标3**：试衣完成率>50%，平均等待时长<3分钟
- **目标4**：支持云端历史记录同步和付费会员体系

### 衡量指标
| 指标名称 | 定义 | 目标值 |
|----------|------|--------|
| 试衣完成率 | 上传图片 → 成功生成结果的比例 | >50% |
| 平均等待时长 | 从点击"开始试衣"到结果展示的时间 | <3分钟 |
| 图片上传失败率 | 图片上传失败的比例 | <5% |
| 历史记录使用率 | 使用历史记录功能的用户比例 | >20% |
| 登录转化率 | 游客转注册用户的比例 | >15% |
| 付费转化率 | 免费用户转付费用户的比例 | >3% |

## 3. 用户流程

### 3.1 核心场景

**场景一：首次使用 - 游客模式试衣**
> **角色**：游客用户  
> **场景描述**：用户首次访问，无需登录即可免费试用3次  
> **步骤**：
> 1. 用户首次访问，自动分配设备ID（localStorage）
> 2. 进入首页，看到品牌展示和两个上传区域
> 3. 点击"上传人物照片" → 选择并上传照片
> 4. 点击"上传衣服图片" → 选择并上传图片
> 5. （可选）展开高级选项选择服装类型（上装/下装/连体装）
> 6. 点击"✨ 开始试衣"
> 7. 等待加载，展示通用提示文案
> 8. 生成成功，展示试衣效果
> 9. （可选）保存图片 / 重新试衣
> 10. 次数用完后，提示登录或升级会员

---

**场景二：用户登录与账户同步**
> **角色**：注册用户  
> **场景描述**：用户登录后，游客记录合并至账户，跨设备同步历史  
> **步骤**：
> 1. 点击"我的"页面或"升级会员"
> 2. 选择登录方式：手机号验证码登录
> 3. 输入手机号，获取验证码
> 4. 输入验证码，登录成功
> 5. 系统自动合并游客本地记录至账户
> 6. 登录状态保持7天（token refresh）

---

**场景三：查看云端历史记录**
> **角色**：注册用户  
> **场景描述**：用户查看云端历史记录，支持跨设备同步  
> **步骤**：
> 1. 点击底部"历史"Tab
> 2. 分页加载历史记录（每页20条）
> 3. 点击某条记录查看详情
> 4. （可选）删除单条/批量删除
> 5. （可选）清空全部

---

**场景四：升级会员（付费）**
> **角色**：免费用户  
> **场景描述**：用户升级为畅用版会员，解除次数限制  
> **步骤**：
> 1. 点击"升级会员"按钮
> 2. 选择套餐：月卡¥9.9/年卡¥49
> 3. 跳转支付页面（支付宝/微信沙箱）
> 4. 支付成功，自动更新会员权益
> 5. 每日次数限制解除

---

**场景五：管理员后台 - 数据查看与管理**
> **角色**：系统管理员  
> **场景描述**：管理员查看系统数据和进行基础管理
> **步骤**：
> 1. 登录后台
> 2. 查看仪表板数据概览
> 3. 查看/管理用户、订单、试衣记录、照片
> 4. 配置系统参数

### 3.2 试衣流程状态机（异步任务+轮询）

| 当前状态 | 触发动作 | 下一状态 | 系统响应 |
|----------|----------|----------|----------|
| 未上传 | 上传人物照片 | 人物已上传 | 显示人物照片预览 |
| 人物已上传 | 上传衣服图片 | 准备就绪 | 显示衣服照片预览，启用"开始试衣"按钮 |
| 准备就绪 | 点击"开始试衣" | 任务提交中 | 提交任务，获取task_id |
| 任务提交中 | 获取task_id成功 | 轮询中 | 开始轮询任务状态 |
| 轮询中 | 状态为queued/processing | 轮询中 | 显示通用提示文案，每5秒更换 |
| 轮询中 | 状态为completed | 已生成 | 展示试衣效果，保存记录 |
| 轮询中 | 状态为failed | 准备就绪 | 显示错误提示，允许重试 |
| 轮询中 | 轮询超时（5分钟） | 准备就绪 | 提示稍后到历史记录查看 |
| 已生成 | 点击"重新试衣" | 准备就绪 | 清除结果，保留上传照片 |

## 4. 功能详细说明

### 4.0 用户登录与账户管理

**前置条件**：无

**UI元素**：
- 首页底部浮层或"我的"页面内提供登录入口
- 登录方式：手机号验证码登录
- 未登录时显示"游客模式"，部分功能受限（如不能查看云端历史、不能付费）
- 登录状态保持7天

**交互逻辑**：
1. 用户首次访问自动分配设备ID（localStorage），作为游客标识
2. 游客可免费试用3次，但次数仅存储于本地
3. 点击"升级会员"或"查看云端历史"时引导登录
4. 登录后，游客记录合并至账户下
5. JWT token有效期2小时，refresh token有效期7天

**异常逻辑**：
- 手机号格式错误：提示"请输入有效手机号"
- 验证码错误：提示"验证码错误，请重新输入"
- 网络超时：Toast提示"网络不稳定，请稍后重试"

**权限**：付费用户需要登录后识别

### 4.1 功能模块一：首页 - 简洁版上传

**前置条件**：无（游客/登录用户均可使用）

**UI元素**：
- 顶部品牌区域：渐变玫红色背景，品牌Logo + 应用名称 + 标语
- 今日剩余次数：右上角，图标+数字
- 上传人物照片区域：圆角矩形，虚线边框，上传图标+文字
- 上传衣服图片区域：同上
- 高级选项：默认折叠，点击展开
- 开始试衣按钮：全宽渐变按钮
- 底部Tab导航：首页/历史/我的
- 底部隐私提示："图片将加密存储30天以便您查看历史，您可随时删除。"

**交互逻辑**：
1. 点击上传区域 → 弹出文件选择器
2. 选择图片后，前端验证：
   - 格式：JPG/PNG/WEBP
   - 大小：≤10MB
3. 验证通过后，上传至云端OSS，获取URL
4. 两张都上传后，启用"开始试衣"按钮
5. 点击"高级选项"→ 展开服装类型选择
6. 点击"开始试衣"→ 提交异步任务

**异常逻辑**：
- 图片格式错误：Toast提示"请上传JPG/PNG/WEBP格式图片"
- 图片过大：Toast提示"图片大小不能超过10MB"
- 未上传完整：点击按钮提示"请先上传人物照片和衣服图片"
- OSS上传失败：Toast提示"图片上传失败，请重试"

**权限**：所有用户可见，游客每日3次，登录用户根据会员权益

### 4.2 功能模块二：试衣效果生成（异步任务+轮询）

**前置条件**：已上传两张图片

**UI元素**：
- 加载动画（旋转圆环）
- 通用提示文案（每5秒更换）
- 超过5分钟提示"生成时间较长，可稍后到历史记录中查看"

**交互逻辑**：
1. 点击"开始试衣"→ 提交人物图URL、服装图URL、类别参数至后端
2. 后端返回task_id
3. 前端每2秒轮询GET /api/tryon/status/{task_id}
4. 轮询返回状态：queued、processing、completed、failed
5. 轮询总超时5分钟
6. 任务完成后自动展示结果图
7. 结果图自动保存至云端历史记录

**异常逻辑**：
- 网络超时：Toast提示"网络不稳定，请稍后重试"
- 任务失败：Toast提示"生成失败，请检查图片是否为清晰的正面人物照片和完整服装图片"
- 轮询超时：提示"生成时间较长，可稍后到历史记录中查看"

**权限**：所有用户可见

## 5. 非功能需求

### 5.1 性能
- 页面首屏加载时间 < 2s
- 图片上传响应时间 < 3s
- 支持并发 100 QPS
- 图片预处理：前端压缩，最大宽度1920px
- 历史记录分页加载（每页20条）

### 5.2 兼容性
- **移动端**：iOS 13+，Android 8+
- **桌面端**：Chrome 最新版、Safari 最新版、Firefox 最新版
- **响应式断点**：<768px（移动端）、768-1024px（平板）、>1024px（桌面）

### 5.3 安全与合规
- 图片将加密存储30天以便您查看历史，您可随时删除
- 上传区域提示："请上传清晰的正面人物照片和完整服装图片"
- 历史记录提供"删除单条/批量删除/清空所有记录"按钮
- 用户注销时自动删除所有图片
- 图片数据传输使用HTTPS
- 后台密码使用bcrypt加密存储
- JWT token有效期2小时，refresh token有效期7天
- 未登录游客每日3次限制，清除localStorage不会增加次数（后端记录设备ID）

### 5.4 存储策略
- 使用阿里云OSS/七牛云存储图片，数据库中存储相对路径
- 表user_photos、tryon_records中图片字段为URL
- 用户上传的人物照片和衣服图片以及生成的试衣效果图，加密存储于云端30天
- 30天后自动物理删除
- 不会将图片用于模型训练或向第三方提供

### 5.5 埋点（可选，V1.0暂不做）
| 事件名 | 触发时机 | 参数 |
|--------|----------|------|
| upload_click | 点击上传按钮 | { type: person/cloth } |
| tryon_start | 点击开始试衣 | { category } |
| tryon_success | 生成成功 | { duration } |
| tryon_fail | 生成失败 | { error_code } |
| save_click | 点击保存图片 | - |
| history_view | 查看历史 | - |
| login_click | 点击登录 | { method } |
| pay_click | 点击升级会员 | { plan } |

| 图片存储成本过高 | 中 | 中 | 30天自动清理，使用压缩图，控制存储成本 |

### 8.4 部署方案（零运维、近乎零成本）

#### 🎯 推荐架构
**最佳方案**：Vercel (前端) + Railway (后端+数据库+Redis) + Cloudflare R2 (图片存储)

**核心优势**：
- 完全免服务器管理：无需配置 Nginx、Docker、K8s，代码推送即部署
- 成本极低：所有服务都有慷慨的免费额度，足够支撑初期几百甚至上千用户
- 自动伸缩：流量增加时自动扩容，无需人工干预
- 国内访问尚可：Vercel 和 Railway 在国内速度中等，但可接受；如需优化可后续加一层 CDN

---

#### 📦 各组件部署详细步骤

##### 1. 前端（React + Vite）→ 部署到 Vercel
**为什么选 Vercel**：对前端框架支持最好，Git 集成，自动构建，免费 CDN。

**操作步骤**：
1. 将前端代码推送到 GitHub 仓库（私有即可）
2. 登录 vercel.com（用 GitHub 账号）
3. 点击 "Add New Project" → 导入你的前端仓库
4. 构建配置一般自动识别（Vite 项目需设置输出目录为 dist）
5. 点击 Deploy → 30 秒后获得 xxx.vercel.app 域名
6. **环境变量**：在 Vercel 项目设置中添加后端 API 地址（部署 Railway 后获得）

**前端环境变量示例（Vercel 中设置）**：
```
VITE_API_BASE_URL=https://your-backend.railway.app
```

---

##### 2. 后端（NestJS + Prisma + PostgreSQL + Redis）→ 部署到 Railway
**为什么选 Railway**：一键部署 Node.js 项目，自动提供 PostgreSQL、Redis 等附加组件，免费额度 $5/月，足够初期使用。

**操作步骤**：
1. 将后端代码推送到 GitHub 仓库
2. 登录 railway.app（用 GitHub 账号）
3. 点击 "New Project" → "Deploy from GitHub repo" → 选择你的后端仓库
4. Railway 会自动检测到 package.json 并部署（执行 npm install && npm run start:prod）
5. 添加数据库：点击项目中的 "New" → "Database" → 选择 "PostgreSQL"。Railway 会自动注入 DATABASE_URL 环境变量
6. 添加 Redis：同样 "New" → "Database" → 选择 "Redis"。会自动注入 REDIS_URL
7. 运行数据库迁移：在 Railway 的 "Shell" 中运行 `npx prisma migrate deploy`（或设置部署后命令）
8. 设置其他环境变量：如 JWT 密钥、云存储配置等（通过 Dashboard 的 Variables 添加）
9. 部署完成后，Railway 会生成一个 *.railway.app 域名，作为后端 API 地址

**后端环境变量示例（在 Railway 中设置）**：
```
DATABASE_URL=postgresql://... (Railway 自动生成)
REDIS_URL=redis://... (Railway 自动生成)
JWT_SECRET=your-strong-secret
JWT_REFRESH_SECRET=another-secret
R2_ACCESS_KEY_ID=your-r2-key
R2_SECRET_ACCESS_KEY=your-r2-secret
R2_BUCKET_NAME=virtual-tryon
R2_PUBLIC_URL=https://pub-xxx.r2.dev
FASHN_API_URL=https://miaoshaohua--fashn-vton-api-fastapi-app.modal.run
```

**数据库连接池问题（重要）**：
- Railway 的 PostgreSQL 默认支持连接池（通过 DATABASE_URL 的 ?pgbouncer=true 参数）
- 在 Prisma 的 schema.prisma 中设置 connectionLimit 为 10~20 即可

---

##### 3. 图片存储（用户上传的人物/衣服图 + 生成的结果图）→ 部署到 Cloudflare R2
**为什么选 R2**：兼容 S3 协议，免费额度 10GB 存储 + 每月 1000 万次请求，无需备案，全球 CDN 加速。

**操作步骤**：
1. 注册 Cloudflare 账号，进入 R2 控制台
2. 创建一个私有桶（例如 virtual-tryon）
3. 在 "设置" 中允许公开访问（生成公开 URL）
4. 创建 API Token（Access Key 和 Secret Key）
5. 在后端代码中配置 AWS SDK（或类似库）连接 R2，上传图片后返回公开 URL 存入数据库

**图片上传流程优化**：
- 前端直接上传到 Cloudflare R2 的预签名 URL（避免经过后端，节省带宽）
- 或者由后端代理上传（简单但会占用后端资源）。建议采用前端直传 R2，后端只负责生成预签名 URL 和存储记录。

**注意**：你需要在后端设置 CORS 规则，允许前端跨域上传。

---

##### 4. 支付功能（沙箱/正式）
**建议**：V1.0 先用模拟支付（点击"支付"直接成功，记录订单），等产品验证有真实付费需求后，再申请支付宝/微信支付企业资质。

如果你已有企业资质，可接入支付宝沙箱或微信支付沙箱，部署到 Railway 的后端可正常调用。

#### ⚠️ 特别提醒（一人公司避坑）

1. **支付功能先模拟**：正式接入支付宝/微信需要企业营业执照，一人公司可以用"联系客服升级"或"虚拟货币"替代，等有收入再注册个体户。

3. **图片合规**：必须在用户协议中明确"图片加密存储 30 天，用户可随时删除"，并在后台提供照片管理功能。

4. **后端数据库备份**：Railway 会自动备份，但你也可以每周手动导出一次 PostgreSQL 数据（免费）。
