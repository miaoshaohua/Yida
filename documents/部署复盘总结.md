## 二、典型问题清单与解析（重点）

### 问题 1：Git 克隆失败 — 连接超时

**现象：**
```
fatal: unable to access 'https://github.com/miaoshaohua/Yida/': 
Failed to connect to github.com port 443: Connection timed out
```

**原因：** 你的腾讯云服务器在国内，GitHub 服务器在国外，网络不稳定。就像你想打电话给国外朋友，但信号塔信号不好。

**解决方法：**
- 使用镜像加速：`git clone https://ghproxy.com/https://github.com/...`
- 或者等待网络好的时候再执行 `git pull`

**避免下次：**
- 服务器放在国外（如香港）可以避免这个问题
- 或者使用 Gitee、Coding 等国内 Git 服务作为镜像

---

### 问题 2：docker-compose.yml 格式错误 — 缩进不对

**现象：**
```
yaml: line 79: did not find expected key
```

**原因：** YAML 文件对缩进非常严格，就像 Python 一样。`backend:` 前面少了两个空格，Docker 就看不懂了。

**解决方法：** 重新创建正确的 docker-compose.yml 文件。

**避免下次：**
- 使用编辑器打开时，注意看缩进标记
- 用 `docker-compose config` 可以先验证格式

---

### 问题 3：环境变量未设置 — 密码不匹配

**现象：**
```
Authentication failed against database server, 
the provided database credentials for `yida` are not valid.
```

**原因：** `.env.prod` 里写的密码是 `Yida2026!SecurePass`，但 PostgreSQL 容器创建时用的是默认密码 `yida123`。就像你拿着 A 钥匙去开 B 锁。

**解决方法：** 让两者保持一致，修改 `.env.prod`：
```bash
POSTGRES_PASSWORD=yida123
```

**避免下次：**
- 创建容器前，先确认 `.env` 文件的值
- 用 `docker-compose config` 查看实际生效的配置

---

### 问题 4：数据库表不存在 — 后端不断重启

**现象：**
```
The table `public.Admin` does not exist in the current database.
```
后端容器启动 → 报错 → 重启 → 报错 → 重启（死循环）

**原因：** 数据库创建了，但里面还没有表。后端启动时要查表，发现表不存在就崩溃了。就像你盖了一栋楼，但里面还没装修，住户住不进去。

**解决方法：**
```bash
# 进入后端容器执行数据库迁移
docker exec yida-backend npx prisma db push
```

**避免下次：**
- 在项目启动脚本中自动执行 `prisma db push`
- 或者使用 Prisma Seed 自动初始化数据

---

### 问题 5：前端 crypto.subtle 报错 — 游客登录失败

**现象：**
```
TypeError: Cannot read properties of undefined (reading 'digest')
```

**原因：** `crypto.subtle.digest` 这个加密函数，只能在 HTTPS 或 localhost 下使用。你的服务器是 HTTP 访问，浏览器认为"不安全"，不让你用。就像银行要求必须用 U 盾才能转账，但你没有 U 盾。

**解决方法：** 添加备用方案：
```typescript
if (crypto.subtle) {
  // HTTPS 环境：使用 SHA-256
  const hashBuffer = await crypto.subtle.digest('SHA-256', data);
} else {
  // HTTP 环境：使用简单哈希
  let hash = 0;
  for (let i = 0; i < components.length; i++) {
    hash = ((hash << 5) - hash) + char;
  }
}
```

**避免下次：**
- 生产环境必须配置 HTTPS
- 开发时就考虑兼容性，不要依赖浏览器特有 API

---

### 问题 6：Nginx 缺少代理配置 — 405 错误

**现象：**
```
405 Not Allowed
nginx/1.29.8
```

**原因：** 前端页面能打开，但点击试衣时报错。因为 Nginx 只负责提供静态文件，不知道 `/api/xxx` 请求应该转发给后端。就像你去了公司前台，但前台不知道把客人引到哪个部门。

**解决方法：** 在 nginx.conf 中添加：
```nginx
location /api/ {
    proxy_pass http://yida-backend:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}
```

**避免下次：**
- 部署前先规划好 Nginx 配置
- 前端调用后端 API 时，确保 Nginx 有对应的 proxy_pass

---

### 问题 7：FASHN_API_URL 包含反引号 — 试衣失败

**现象：**
```
调用FASHN API: `https://miaoshaohua--fashn-vton-api-fastapi-app.modal.run/tryon`
Request failed with status code 401
```

**原因：** `.env.prod` 文件里，URL 被反引号 `` ` `` 包裹了，导致实际值是 `` `https://...` `` 而不是 `https://...`。就像你寄快递，地址写的是"北京"，但实际写成了 `"北京"`，快递公司不认识引号。

**解决方法：**
```bash
sed -i 's/FASHN_API_URL=.*/FASHN_API_URL=https:\/\/miaoshaohua--fashn-vton-api-fastapi-app.modal.run/' .env.prod
```

**避免下次：**
- `.env` 文件中不要使用引号包裹值
- 用 `cat .env.prod | grep FASHN` 验证配置是否正确

---

### 问题 8：管理员密码为空 — 无法登录后台

**现象：** 管理员账号存在，但 `password` 字段是空的，无法登录。

**原因：** 后端初始化超级管理员时，可能因为数据库连接问题，密码没有正确写入。

**解决方法：**
```bash
# 用后端容器生成正确的密码哈希
HASH=$(sudo docker exec yida-backend node -e "const bcrypt = require('bcryptjs'); console.log(bcrypt.hashSync('admin123', 10));")
# 更新数据库
sudo docker exec -e PGPASSWORD=yida123 yida-postgres psql -U yida -d yida -c "UPDATE \"Admin\" SET password = '${HASH}' WHERE email = 'admin@yida.com';"
```

**避免下次：**
- 在部署文档中明确记录初始管理员账号密码
- 添加健康检查接口验证管理员登录

## 三、知识盲区扫盲

### 1. 公网 IP

**是什么：** 互联网上唯一标识你服务器的地址。

**通俗理解：** 就像你家的门牌号。快递小哥（用户）必须知道这个地址，才能把包裹（请求）送到你家（服务器）。

**你的情况：** `42.194.162.239` 就是你服务器的公网 IP。

---

### 2. 端口

**是什么：** 一台服务器上可以运行多个服务，端口用于区分它们。

**通俗理解：** 就像一栋大楼有很多房间，IP 是大楼地址，端口是房间号。

| 端口 | 服务 | 用途 |
|------|------|------|
| 80 | Nginx | 提供前端网页 |
| 3000 | NestJS | 提供 API 接口 |
| 5432 | PostgreSQL | 数据库服务 |
| 6379 | Redis | 缓存服务 |

---

### 3. 进程守护

**是什么：** 让程序在后台持续运行，即使你关闭 SSH 连接也不会停止。

**通俗理解：** 就像你雇了一个保姆，你出门后她继续在家工作。

**你的情况：** `restart: unless-stopped` 就是进程守护配置，容器崩溃后会自动重启。

---

### 4. 反向代理

**是什么：** 接收用户的请求，然后转发给内部的服务，再把结果返回给用户。

**通俗理解：** 就像公司前台。客户来找人，前台问清楚后，把客户引到正确的部门。

**你的情况：** Nginx 就是反向代理，它把 `/api/xxx` 请求转发给后端 `yida-backend:3000`。

---

### 5. 环境变量

**是什么：** 程序运行时需要的外部配置信息，不写在代码里。

**通俗理解：** 就像你去酒店入住，房卡（代码）是通用的，但开哪个房间（配置）取决于前台给你什么钥匙卡（环境变量）。

**你的情况：** `.env.prod` 文件就是环境变量配置，包含数据库密码、API 密钥等敏感信息。

## 五、下一步学习建议

### 1. 配置 HTTPS（最紧迫）🔥

**为什么：** 你的网站目前是 HTTP，浏览器会显示"不安全"警告，而且某些浏览器 API（如 `crypto.subtle`）无法使用。

**怎么做：**
```bash
# 安装 Certbot（免费 SSL 证书）
apt install -y certbot python3-certbot-nginx

# 获取证书（需要域名）
certbot certonly --standalone -d yourdomain.com
```

**学习时间：** 2-3 小时

---

### 2. 学习基本 Linux 命令

**为什么：** 部署过程中，很多操作都需要命令行。掌握基础命令能让你更自信。

**推荐学习：**
```bash
# 文件操作
ls, cd, cat, cp, mv, rm, mkdir

# 进程管理
ps, top, kill, systemctl

# 网络相关
netstat, curl, ping

# 日志查看
tail -f, grep, journalctl
```

**学习资源：** B 站搜索"Linux 基础命令"，看 2-3 小时视频即可。

---

### 3. 理解 Docker 核心概念

**为什么：** 你已经用 Docker 部署成功了，但可能还不完全理解每个命令的含义。

**推荐学习：**
- Dockerfile 是什么？怎么写的？
- docker-compose.yml 的结构
- 数据卷（Volume）的作用
- 网络（Network）的配置

**学习资源：** Docker 官方文档的 "Get Started" 部分，约 3-5 小时。
